# Cepheus Processors

Given that cepheus is  a distributed system, the processors themselves are a separate service defined by the type of data they consume and process. Currently, we have three processors: `stamp-processor` for STAMP probes, `ping-processor` for PING probes and `trace-processor` for processing `scamper`-generated traces ( `trace` or `tracelb`).

### Why are processors independently managed per-probe type ?
The decision comes from the fact that, in comparision to **STAMP** or **PING** data, traceroute data requires more processing, hence different scability requirements. These "extra" processing steps include things like enriching ASN data, calculating fingerprints, and de-duplication.

## Ingestion lifecycle
Each processor polls for a `cepheus.probe.<processor_type>.>` subject from NATS Jetstream server. It ingests the data, processes it, and writes it to the database. 

## Specifications for a new processor
Each agent sends a probe data payload in the format specified below. The `ReportPayload` contains information gathered during dispatch. It encloses the `ProbeResult` that is generated by the probe worker in the agent. `ProbeResult.Payload` contains the actual data for processing. `ProbeResult` also captures necessary metadata when packing the probe data, before being sent to the dispatcher.

All processors must follow this guidelines and deseralize data based on the this probe packet diagram.

> [!NOTE]
> I understand that the abstraction in the payload packet is complicated without benefits. We will redesign this to be much flatter to decrease the number of serialization/deserialization steps in the future.

```
+---------------------------------------------------+                                           
|Report Payload                                     |                                           
|                                                   |                                           
|  +----------------------------------------------+ |                                           
|  |SerialId      string                          | |                                           
|  +----------------------------------------------+ |                                           
|  |SentTimestamp time.Time                       | |                                           
|  +----------------------------------------------+ |                                           
|  |Payload       common.ProbeResult              | |                                           
|  |                                              | |                                           
|  |  +-----------------------------------------+ | |                                           
|  |  | Data      json.RawMessage     <---------+-+-+------- Payload that the processor requires
|  |  |                                         | | |        [ For example:                     
|  |  | ...                                     | | |              - PingDataPayload

```

### Future considerations
Currently, the processors directly write the data to the DB, and the detection engine, Argus, reads from the database, maintains a state of data lookups and run the detection algorithm. We could improve on this by directly sending the processed data to the detection engine through either NATS or a separate Kafka instance, and make DB writes asynchronous.
